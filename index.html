9.0

Introduction — Why I/O Matters

When you buy a computer, you probably look at how fast the CPU is. But no matter how powerful the CPU is, the computer is totally useless without Input/Output (I/O). If you can't get data into the computer (keyboard, mouse, network) and you can't see the results (screen, printer, disk), the CPU is just doing math in the dark.

🔑 Core Problem

The CPU is incredibly fast, processing billions of operations a second. I/O devices, however, are slow because they often rely on physical mechanics (like a user typing or a disk spinning). The big challenge in computer architecture is keeping the lightning-fast CPU from constantly pausing to wait on slow I/O devices.

🔑 The Real Bottleneck

For most modern programs—like web browsing, playing videos, or running a business database—the CPU spends very little time actually "thinking". The dominant factor holding back performance is how quickly data can be fetched from the hard drive or network. I/O, not CPU speed, is the real bottleneck.

📖 Little Man Computer Analogy
Think back to the Little Man Computer (LMC) model. The "In" and "Out" baskets were the I/O system. The Little Man could only take one 3-digit number at a time, requiring a specific instruction just to grab that one number. Early computers worked exactly like this, but as data demands grew (imagine reading a 4-gigabyte movie one number at a time), we had to invent faster I/O methods like DMA.
9.1

Characteristics of Typical I/O Devices

To design a good I/O system, engineers first had to look at what different devices actually demand. A keyboard and a graphics card behave entirely differently and require different management from the CPU.

Device In/Out Data Rate Transfer Mode
Keyboard / Mouse Input Very Low Character (One by one)
Printer Output Low–Medium Block burst
Magnetic Disk / SSD Storage Medium–High Block burst
Graphics Display Output High Steady stream
Network / USB Both High–Very High Block burst

Why Device Differences Matter

1. Expected vs. Unexpected Input (Keyboards & Mice)

Sometimes a program explicitly waits for you to type something, like entering your password. This is "expected input." However, if a program freezes and you hit `Ctrl+Alt+Delete` to force close it, the CPU was not expecting that. Because devices like keyboards and mice frequently generate unexpected input, the I/O system must have a way to interrupt whatever the CPU is currently doing to pay attention immediately.

2. Stream vs. Burst (Displays vs. Printers)

A printer accepts data in chunks (blocks), prints a page, and if the data stops coming, it just waits patiently. A computer monitor playing a 60fps video is different. It requires a continuous, steady stream of heavy data. If the data stops for even a millisecond, the screen flickers or the video stutters. The I/O system has to prioritize steady stream devices differently than burst devices.

3. Blocks vs. Characters (Disks vs. Keyboards)

When you press the letter "A" on a keyboard, a single character (usually 1 byte) is sent to the CPU. Hard drives and SSDs never do this. If you want to change one single letter in a text document saved on a drive, the computer must read an entire block of data (often 4,000+ bytes), change the one letter in memory, and write the whole massive block back. Disks transfer massive bursts of block data.

💡 The Role of Buffers
Because a CPU is so fast and a printer is so slow, if they talked directly, the CPU would have to drastically slow down to match the printer's speed. Instead, we use a buffer—a temporary holding area. The CPU blasts the data into the buffer instantly, then goes back to doing other work. The printer then slowly pulls the data out of the buffer at its own pace. Think of it like a water tower: the pump fills it up fast, and houses drain it slowly over time.
9.2

Programmed I/O

Programmed I/O is the oldest and simplest way a computer communicates with a device. In this method, the CPU is the sole worker. It manually handles every single piece of data, moving it step-by-step from the device, into its own registers, and then into memory. It requires a full instruction to execute for every single piece of data transferred.

The "Teaspoon" Analogy

Imagine you have to move a giant pile of sand (a file on a hard drive) from the yard (the disk) into a sandbox (the RAM). Programmed I/O is like using a teaspoon to move the sand.

The CPU takes a teaspoon of sand (one byte), walks it over, drops it in the sandbox, goes back, and repeats this millions of times. While the CPU is doing this, it is 100% occupied. It cannot run your video game, play music, or check for network messages. It is totally locked into this tedious task.

✅ When is Programmed I/O Good?

It is perfectly fine for slow, low-volume devices like keyboards. Since humans type slowly (maybe a few characters a second), moving one "teaspoon" of data every so often doesn't burden the CPU at all. It is also used to send short command instructions to initialize hardware.

❌ When is it Terrible?

It is unusable for transferring files from a hard drive or feeding graphics to a monitor. Moving megabytes of data one word at a time would bring the entire computer to a crawling halt. For these, we must use DMA (covered in 9.4).

9.3

Interrupts

If Programmed I/O is bad for unexpected events, how does the CPU know when something happens? It uses Interrupts. An interrupt is literally an electronic signal sent on a physical wire to the CPU, telling it: "Stop what you are doing right now, save your place, and pay attention to me!"

🎓 Classroom Analogy
Imagine you are a teacher giving a lecture to a class (the CPU executing a program). A student raises their hand and interrupts with a question. You don't abandon the whole lecture; you hold your thought, answer the student's question (handle the interrupt), and then resume your lecture exactly where you left off. This is how normal hardware interrupts work.

The Concept of "Context"

When the CPU is interrupted, it has to remember exactly what it was doing so it can go back to it later. It does this by saving its Context. The context consists of all the current values in the CPU's registers, and most importantly, the Program Counter (which tracks the next instruction). It saves this context to a special area in memory (a Stack or a Process Control Block), runs the Interrupt Service Routine (ISR) to handle the hardware request, and then restores the context to resume normal operations.

The 4 Major Uses of Interrupts

Interrupts aren't just for typing on a keyboard. The entire operating system relies on them for four main purposes:

  1. External Event Notifier (Getting Attention): Instead of the CPU constantly wasting time asking the keyboard "did you press a key? did you press a key?" (a process called polling), the CPU ignores the keyboard until the keyboard fires an interrupt saying "Hey, I have a keystroke!". This saves massive amounts of CPU time.
  2. Completion Signal (Done Processing): When the CPU asks the hard drive to fetch a large file, the hard drive takes a long time to mechanically find it. The CPU goes and works on other programs. When the hard drive finally has the data ready in its buffer, it sends an interrupt: "I'm done, the data is ready."
  3. CPU Time Allocation (Multi-tasking): How can a computer run Discord, Chrome, and Spotify at the same time if it only has one core? The system clock sends an interrupt every few milliseconds. When the interrupt hits, the OS pauses Chrome, gives CPU time to Discord, pauses Discord, gives time to Spotify, and so on. This forced switching happens so fast it looks simultaneous.
  4. Abnormal Event Indicator (Crashing Gracefully): If a program tries to divide a number by zero, or if a user unplugs a USB drive while data is writing, the hardware generates an error interrupt (often called a Trap or Exception). This allows the OS to step in and shut down just that specific program, rather than letting the whole computer crash.

How to find out who interrupted? (Vectored vs. Polled)

If multiple devices are plugged into the computer, and the CPU receives an interrupt signal, how does it know who sent it?

  • Vectored Interrupt: The device sends the interrupt signal *and* attaches a little ID tag (an address) along with it. The CPU looks at the ID tag and immediately knows "Ah, this is the mouse!" and jumps straight to the mouse driver software. This is very fast but requires complex circuitry.
  • Polled Interrupt: A generic alarm goes off. The CPU doesn't know who did it, so it literally has to go down a line and ask each device: "Was it you? Was it you?" This requires simpler hardware, but it is much slower because of the interrogation process.
9.4

Direct Memory Access (DMA)

Remember the "teaspoon" analogy from Programmed I/O? Direct Memory Access (DMA) is the dump truck. Because moving data one byte at a time through the CPU is too slow for large files or graphics, engineers created a way for devices to bypass the CPU entirely and write data directly into the system's RAM.

How DMA works (The Dump Truck Analogy)

In DMA, the CPU acts as the supervisor, not the worker. Here is how a file transfer from a hard drive works using DMA:

  1. The CPU uses simple Programmed I/O to send a quick "work order" to the Hard Drive Controller. The order contains 4 things: the memory address to drop the data, where to find it on the disk, how big the file is, and whether to read or write.
  2. The CPU steps away. It goes off to run other programs, completely ignoring the file transfer.
  3. The Hard Drive Controller takes over the system bus. It fetches the massive blocks of data and shovels them directly into the RAM (the sandbox).
  4. When the Hard Drive Controller is completely finished, it fires a Completion Interrupt to tap the CPU on the shoulder, saying "The sand is in the sandbox, boss."
⚠️ The Danger of DMA
Because the CPU is busy doing other things while the hard drive is secretly writing data directly into the RAM, a conflict can happen. If the CPU tries to read that exact same section of RAM before the hard drive is finished, it might read half-old data and half-new data, causing a system crash. The OS must lock that section of memory so the CPU can't touch it until the completion interrupt arrives.

Why Interrupts are Required for DMA

You can clearly see why DMA and Interrupts are a team. Without the completion interrupt at the very end of the process, the CPU would never know when the dump truck finished delivering the data. It would just sit around blindly waiting, totally defeating the purpose of being freed up to do other work!

9.5

I/O Controllers

The CPU is a genius mathematician, but it knows absolutely nothing about the physical world. It doesn't know how to spin a magnetic platter, how to squirt ink onto paper, or how to translate color codes into light on an LCD panel. I/O Controllers act as the intelligent translators between the CPU's mathematical world and the physical mechanics of the devices.

The Translator's Two Faces

Every I/O Controller (like a Graphics Card or a Network Interface Card) has two distinct interfaces because it is speaking two completely different languages:

  • The System Side (Talking to the CPU): On this side, the controller speaks the CPU's language. It connects to the motherboard's bus, accepts memory addresses, reads binary commands, and generates standard interrupt signals.
  • The Device Side (Talking to the Hardware): On this side, the controller handles the messy physics. If it's a disk controller, it sends electrical voltages to start a physical motor, calculates the exact timing to drop a read/write head onto a spinning magnetic platter, and handles error correction if there's dust on the disk.

Why We Use Separate Controllers

If we tried to build all this logic directly into the CPU, the CPU would be massive, incredibly expensive, and impossible to upgrade. By separating the logic into dedicated controllers (like buying a separate Nvidia graphics card), we get several benefits:

  1. CPU Freedom: The CPU passes off complex tasks (like rendering 3D shadows) to the Graphics Controller, freeing itself up to run the game's physics and AI logic.
  2. Parallel Operation: Because every device has its own mini-brain (controller), your computer can download a file from the network, spin the hard drive to save it, and play music out of the speakers, all at the exact same time.
  3. Specialization: Companies can design specialized processors just for graphics (GPUs) or just for network traffic, optimizing them far better than a general CPU ever could.

Key Terms Glossary

Programmed I/O
CPU transfers data one word at a time under full program control. One instruction per data word.
Interrupt
A signal on a dedicated line (IRQ) that causes the CPU to suspend its current program and jump to an ISR.
Interrupt Service Routine (ISR)
Special program executed when an interrupt fires. Also called interrupt handler, interrupt routine, or device driver.
Context
All register values and the PC that define a program's current state. Saved on interrupt, restored on return.
Process Control Block (PCB)
Memory area where a program's context is saved during an interrupt.
Polling
CPU continuously checks a device in a loop to see if it needs attention. Wastes CPU cycles. Inferior to interrupts.
Vectored Interrupt
Interrupt includes the device's address. CPU jumps directly to the correct ISR. Faster but needs more hardware.
Polled Interrupt
General interrupt fires; CPU polls each device to find which one triggered it. Slower, simpler hardware.
Trap / Exception
An interrupt generated internally by the CPU (illegal instruction, divide-by-zero, memory error).
DMA (Direct Memory Access)
Transfer of block data directly between I/O controller and memory, bypassing the CPU during transfer.
Buffer
Temporary storage that absorbs the speed difference between two components (like a water reservoir).
Device Controller
I/O controller dedicated to one type of device (disk controller, NIC, graphics card). Has two interfaces: CPU-side and device-side.

Likely Exam Questions & Answers

Q: Why is polling inefficient? Polling forces the CPU to continuously loop-check a device, wasting millions of CPU cycles doing nothing useful. Interrupts let the CPU do real work and only react when the device actually needs attention.
Q: What is a context, what does it contain, and why is it saved? Context = the complete state of a program: all register values + the program counter (PC). Saved to the stack or PCB when an interrupt occurs. Restored when the ISR finishes — so the original program resumes exactly as if nothing happened.
Q: Why would DMA be useless without interrupt capability? Without a completion interrupt, the CPU has no reliable way to know when the DMA transfer is finished. It would have to poll the controller (inefficient), or wait blindly. The whole benefit of DMA — CPU freedom during transfer — depends on the completion interrupt to signal when data is safely in memory.
